# Project Overview

This is a Shopify theme project built on **Dawn** (Shopify's official reference theme). We customize Dawn to match client requirements — adding new sections, overriding styles, extending functionality — without breaking the upstream theme structure.

---

## Stack

| Layer | Technology |
|---|---|
| Base Theme | Dawn (latest stable) |
| Template Language | Liquid |
| Styles | Tailwind CSS v4 + Dawn CSS variables |
| JS | Vanilla JS + Web Components (Dawn's pattern) |
| CSS Build | `@tailwindcss/cli` — compiles `css/application.css` → `assets/application.css.liquid` |
| Code Formatter | Prettier + `@shopify/prettier-plugin-liquid` |
| JS Linter | ESLint (`@eslint/js`) — `eslint.config.mjs` |
| Pre-commit Hooks | Husky + lint-staged |
| Theme Linter | Shopify Theme Check |
| CLI | Shopify CLI 3.x |
| Version Control | Git |
| Package Manager | npm |

## Development Philosophy

- **Customize, don't fork** — Override Dawn's CSS variables and extend sections. Avoid rewriting Dawn's core files unless absolutely necessary.
- **Dawn's Web Component pattern** — Dawn uses native Web Components (Custom Elements). Follow this pattern when adding interactive JS behavior.
- **Tailwind v4 for custom UI** — Use Tailwind utility classes for all new custom sections and components. Write source in `css/application.css`, never in `assets/application.css.liquid`.
- **CSS variable first for branding** — Override Dawn's CSS variables for colors, fonts, spacing. Use Tailwind for layout and component-level styling.
- **Section modularity** — Every layout block the client can control via theme editor = a section. Keep sections focused and composable.
- **No jQuery** — Dawn is jQuery-free. Keep it that way.
- **Accessibility** — Maintain Dawn's ARIA patterns. Don't remove focus states or aria attributes.
- **Code quality is automated** — Prettier, ESLint, Husky, and Theme Check run on every commit. Don't bypass them.

## CSS & Tailwind Conventions

### Tailwind v4
- Source file: `css/application.css` — all Tailwind imports and custom CSS go here.
- Output file: `assets/application.css.liquid` — auto-generated, never edit manually.
- Included in `theme.liquid` as: `{{ 'application.css' | asset_url | stylesheet_tag }}`
- Build command: `npm run build` (runs `npx @tailwindcss/cli -i css/application.css -o assets/application.css.liquid -w -m`)
- Use Tailwind utility classes for custom sections, components, and layouts.

```css
/* css/application.css */
@import "tailwindcss";

/* Custom CSS and Dawn variable overrides go below */
```

### Dawn CSS Variables
- Override Dawn's CSS variables for brand colors, fonts, and spacing — they cascade to all Dawn components automatically.
- Dawn uses RGB values WITHOUT commas: `255 255 255` not `#fff`.
- Custom overrides: add them in `css/application.css` below the Tailwind import.
- Use BEM naming for any new non-Tailwind classes: `.block__element--modifier`.
- Never use `!important` unless forced by Shopify's inline styles — document it when used.

### Breakpoints (Dawn's pattern)
- Small / Mobile: up to `749px`
- Tablet+: `750px` and above
- Desktop: `990px` and above

```css
/* Dawn's core CSS variables — override these for branding */
:root {
  --color-base-background-1: 255 255 255;
  --color-base-text: 18 18 18;
  --font-body-family: 'YourFont', sans-serif;
  /* etc. */
}
```

## Liquid Conventions

- Use `{% liquid %}` tags for multi-line logic blocks — keeps templates readable.
- Assign variables at the top of a section before rendering markup.
- Use `{% render %}` (not `{% include %}`) for snippets — it's scoped and faster.
- Always provide fallback values: `{{ section.settings.title | default: 'Default Title' }}`.
- Escape user content: `{{ content | escape }}` where appropriate.
- Comment complex logic: `{% comment %} ... {% endcomment %}`.

---

## Section Schema Conventions

- Every section must have a `name`, `tag`, and `class` in schema.
- Always include `presets` so the section appears in the theme editor "Add section" list.
- Group related settings using `header` type for visual separation in the editor.
- Limit settings to what the client actually needs — don't over-engineer the editor.
- Use `image_picker` + `focal_point` for any image settings.

```json
{
  "name": "Section Name",
  "tag": "section",
  "class": "section-name",
  "settings": [...],
  "blocks": [...],
  "presets": [{ "name": "Section Name" }]
}
```

## JavaScript Conventions

- Follow Dawn's Web Component pattern — extend `HTMLElement`, define with `customElements.define()`.
- Use `connectedCallback()` for initialization logic.
- Scope all query selectors inside the component: `this.querySelector(...)` not `document.querySelector(...)`.
- Use `addEventListener` with named handlers so they can be removed in `disconnectedCallback()`.
- No external JS libraries unless absolutely required and bundle-size justified.
- No jQuery.

### ESLint Rules in Effect (enforced on every commit)
- `no-unused-vars` — never declare variables you don't use.
- `no-console` — remove all `console.log` before committing.
- `no-debugger` — no debugger statements.
- `no-var` — always use `const` or `let`.
- `prefer-const` — use `const` unless reassignment is needed.
- `eqeqeq` — always use `===` / `!==`, never `==` / `!=`.
- `curly` — always use curly braces for `if`, `for`, etc.
- `prefer-template` — use template literals, not string concatenation.
- `no-implied-eval` — no `setTimeout('string', ...)` patterns.
- `max-depth: 4` — max nesting depth of 4 levels.
- `no-shadow` — no variable shadowing.
- `no-undef` — no undeclared variables.

```js
// ✅ Correct pattern
if (!customElements.get('my-component')) {
  customElements.define('my-component', class MyComponent extends HTMLElement {
    connectedCallback() {
      this.button = this.querySelector('[data-action]');
      this.handleClick = this.onClick.bind(this);
      this.button?.addEventListener('click', this.handleClick);
    }

    disconnectedCallback() {
      this.button?.removeEventListener('click', this.handleClick);
    }

    onClick() {
      // logic
    }
  });
}
```

## Dev Workflow & npm Scripts

```bash
# Start Tailwind watcher (run this first — always keep running during dev)
npm run build

# Format all Liquid, CSS, JS files with Prettier
npm run format

# Lint JS files with ESLint
npm run lint

# Initialize Husky hooks (run once after cloning)
npm run prepare
```

### Shopify CLI Commands
```bash
# Start local dev server and sync with store
shopify theme dev --store=your-store.myshopify.com

# Push theme to store
shopify theme push

# Pull latest from store
shopify theme pull

# Run theme check manually
shopify theme check
```

### Pre-commit Hook (runs automatically on every `git commit`)
1. `lint-staged` — ESLint fix on `*.js`, Prettier format on all staged files
2. `shopify theme check` — validates Liquid, schema, and theme structure

> ⚠️ Fix all ESLint and Theme Check errors before committing — the hook will block the commit if they fail.

## DO NOTs

- ❌ Don't edit `assets/application.css.liquid` — it's auto-generated by Tailwind CLI. Edit `css/application.css` instead.
- ❌ Don't edit `node_modules` or `.shopify/` folders.
- ❌ Don't add subfolders inside `assets/`.
- ❌ Don't use `{% include %}` — use `{% render %}` instead.
- ❌ Don't hardcode colors or fonts — use CSS variables or Tailwind tokens.
- ❌ Don't remove Dawn's accessibility attributes (aria-*, role, focus-visible).
- ❌ Don't use jQuery or any heavy JS libraries.
- ❌ Don't commit `settings_data.json` with client-specific live data.
- ❌ Don't modify `config/settings_schema.json` without reviewing existing client settings.
- ❌ Don't use `var` — use `const` or `let`.
- ❌ Don't leave `console.log` or `debugger` statements — ESLint will block the commit.
- ❌ Don't use string arguments in `setTimeout` / `setInterval`.
- ❌ Don't bypass Husky pre-commit hooks with `--no-verify` unless in an emergency — document why.
- ❌ Don't push files listed in `.shopifyignore` (package.json, node_modules, etc.).

## Notes for Claude

- When creating a new section, always scaffold the full file: Liquid markup + `{% schema %}` + `{% stylesheet %}` + `{% javascript %}`.
- For custom section styles, prefer Tailwind utility classes in the Liquid markup. Use `{% stylesheet %}` only for styles that can't be expressed with Tailwind.
- When editing styles, prefer CSS variable overrides over new selectors for Dawn-wide branding changes.
- When in doubt about a Dawn component, check `assets/` for the corresponding JS file — Dawn names them clearly (e.g. `cart-drawer.js`, `product-form.js`).
- Always keep the theme editor experience in mind — sections should be intuitive for non-technical clients.
- Check `sections/main-*.liquid` files to understand how Dawn handles core page layouts before creating custom templates.
- All generated JS must comply with the ESLint rules: no `var`, no `console.log`, use `const`/`let`, use template literals, always use `===`, always use curly braces.
- When writing or modifying JS, keep max nesting depth to 4 levels or less.
- The Tailwind build must be running (`npm run build`) for new utility classes to appear in `assets/application.css.liquid`.
